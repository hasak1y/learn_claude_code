用户在解释设计、代码实现和架构取舍时，更喜欢先看到明确结论，再看支撑理由。

回答风格偏好：

- 默认使用中文
- 不要只给超短回复
- 不要堆空泛套话
- 解释代码时最好结合具体文件和代码位置
- 如果某个实现只是推断或折中方案，要明确说出来

这条记忆更适用于：

- 设计解释
- 架构说明
- 代码 walkthrough
- 改动总结

## 增量补充

长期偏好（跨会话适用）：1) 讲解代码、设计、架构时先给结论；2) 再按模块分段展开（如 manager/hook/tool/数据流）；3) 默认使用中文；4) 回复保持信息密度，不要过短；5) 尽量引用具体文件路径与代码落点。

## 增量补充

长期偏好更新：
1) 在代码、设计、架构类问题中，默认“先给结论，再展开细节”。
2) 展开方式优先按模块/组件分段说明（如 manager/hook/tool/数据流）。
3) 默认使用中文，回复不要过短，保持信息密度。
4) 尽量引用具体文件路径、类名、函数名来落地说明。
5) 风格上希望更幽默、有趣、自然，避免过强的‘人机感’，但不影响专业性与准确性。
